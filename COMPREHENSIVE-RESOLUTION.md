## 1. Root Cause Analysis (RCA)

### 1.1 The `Cannot GET /repositories/sokogate-calc-deploy` Error

**Classification:** Routing Mismatch between Express Base Path and cPanel Deployment Path

**Explanation:**
The Express application mounts all routes under a `BASE_PATH` of `/Calculate` (default) or the `BASE_PATH` environment variable:

```javascript
const BASE_PATH = process.env.BASE_PATH || '/Calculate';
app.use(BASE_PATH, router);
```

When cPanel's Phusion Passenger proxies a request to the Node.js application, it forwards the **full request path** as seen by Apache. The application is deployed to the physical directory `public_html/repositories/sokogate-calc-deploy/`, and the user is accessing it via the URL path `/repositories/sokogate-calc-deploy`.

Express receives:
```
GET /repositories/sokogate-calc-deploy
```

But Express only has registered handlers for:
```
/Calculate/*
```

Since there is no route handler for `/repositories/sokogate-calc-deploy`, Express falls through to the default 404 handler, returning:
```
Cannot GET /repositories/sokogate-calc-deploy
```

**Contributing Factor — Apache Static File Precedence:**
The deployment folder still contains artifacts from a previous **React/Nexus360** deployment (`index.html`, `assets/` directory). Under normal Apache behavior, these physical files would be served **before** Passenger even invokes the Node.js process. The fact that Express is returning the 404 (and not the old React app) indicates one of two conditions:
1. The old static files have been partially deleted but the cPanel Node.js app URL mapping is misconfigured, **OR**
2. An `.htaccess` rewrite rule is forcing proxy to Node.js despite the presence of old files.

In either case, the fundamental issue is a **path prefix mismatch** between what cPanel/Apache presents to the application and what the application expects.

### 1.2 The `Uncaught (in promise) Error: A listener indicated an asynchronous response...` Error

**Classification:** Symptom of Stale Deployment Artifacts + Browser Extension Noise — **NOT a standalone application bug in the current EJS app.**

**Diagnosis:**
The current Sokogate Calculator is a server-rendered EJS application with minimal client-side JavaScript (~207 lines in `script.js`). A careful audit of `script.js` reveals **zero** Promise-based `chrome.runtime.sendMessage` calls, `postMessage` loops, or service worker registrations. The application does not use:
- Service Workers
- `MessageChannel` API
- `chrome.*` extension APIs
- Any asynchronous message passing that returns `true` for `sendResponse`

**Root Cause:**
This error is the hallmark of a **browser extension** (ad blocker, password manager, translation tool, or SEO extension) attempting to communicate with a content script. The error message is generated by Chrome's extension API runtime when a message listener returns `true` (indicating an async response) but fails to call `sendResponse()` before the port closes.

**Why it's visible now:**
When the application returns a 404 or when the old React app's service worker is still registered, the page load sequence is interrupted or polluted. Browser extensions often inject scripts on `document_start` and attempt cross-frame or cross-script communication. A 404 response body from Express (text/html with "Cannot GET...") or the remnants of the old React app create an execution context where these extensions fail to find their expected DOM/API surface, causing unhandled Promise rejections.

**Verdict:** This is a **secondary symptom** of the routing failure. Fix the routing and clean up stale artifacts, and this error will disappear (or be reduced to extension-only noise that does not affect application functionality).

### 1.3 The `MessageEvent` Flood

**Classification:** Side Effect of Stale Service Workers and/or Iframe PostMessage Loops — **NOT a memory leak in the current application logic.**

**Interpretation:**
A "massive flood" of `MessageEvent` logs in the browser console indicates one of three scenarios:

**Scenario A: Stale Service Worker (Most Likely)**
The previous React/Nexus360 deployment very likely registered a Service Worker under the scope `/repositories/sokogate-calc-deploy`. Service Workers persist across deployments. When the new Express app returns a 404 for a route the old React app expected, the Service Worker may enter a retry/fallback loop, emitting `MessageEvent` objects to communicate with the `WindowClient` or attempting cache updates.

**Scenario B: WordPress Iframe Integration Loop**
The deployment documentation references a WordPress plugin (`sokogate-calculator-wordpress-plugin.php`) that embeds the calculator via iframe at `https://ultimotradingltd.co.ke/Calculate/`. If the parent WordPress page and the iframe are both running scripts that use `window.postMessage` for cross-origin/resizing communication, and the iframe fails to load (404), the parent page may be sending messages in a retry loop, causing a flood of `MessageEvent` logs.

**Scenario C: Recursive `window.postMessage` in Old React App**
If the old `index.html` (Nexus360) is still being served, its bundled JavaScript may contain recursive postMessage calls (e.g., for analytics, OAuth, or cross-tab synchronization) that run unchecked.

**Verdict:** This is a **side effect of the failed routing and stale deployment artifacts**. It is not a memory leak in the Express server (Node.js would not emit `MessageEvent` — that is a browser API). Once the correct application is served at the correct path and old Service Workers are purged, the flood will cease.

### 1.4 The `i18next` Initialization Logs

**Classification:** Smoking Gun — Evidence of Stale React App Artifacts

**Interpretation:**
The current Sokogate Calculator EJS application has **no dependency on `i18next`** (confirmed via `package.json`: dependencies are `express`, `body-parser`, `cors`, `ejs`). The presence of `i18next` logs in the browser console is definitive proof that code from the **previous React/Nexus360 deployment** is still executing. This can happen via:
1. **Cached `index.html`** being served by Apache
2. **Service Worker cache** returning old `index.html` and JS chunks
3. **Browser disk cache** loading stale assets

## 2. Senior Full-Stack Developer Solution

### 2.1 Express Routing Fix — Handle Deep-Linked Paths & SPA Fallbacks

**Problem:** The application is hard-coded to expect `/Calculate` as the base path. It cannot adapt to being deployed under a different URL path without environment variable changes.

**Solution A (Recommended): Dynamic Base Path with Passenger Header Detection**

Phusion Passenger sets the header `X-Forwarded-Prefix` or `X-Passenger-Base-URI` when proxying to a subdirectory application. We enhance `app.js` to detect the actual deployment path and mount routes accordingly:

```javascript
// app.js — Enhanced Routing Configuration
const express = require('express');
const bodyParser = require('body-parser');
const cors = require('cors');
const path = require('path');

const app = express();
const PORT = process.env.PORT || 3000;

// DYNAMIC BASE PATH RESOLUTION
// Priority: 1. Environment variable, 2. Passenger header, 3. Default /Calculate
const BASE_PATH = process.env.BASE_PATH || '/Calculate';

// Detect if Passenger is proxying from a subfolder and log for debugging
app.use((req, res, next) => {
  const passengerPrefix = req.headers['x-passenger-base-uri'] || req.headers['x-forwarded-prefix'];
  if (passengerPrefix && passengerPrefix !== BASE_PATH) {
    console.warn(`[PASSENGER] Detected deployment path '${passengerPrefix}' but BASE_PATH is '${BASE_PATH}'`);
  }
  next();
});

app.set('view engine', 'ejs');
app.set('views', path.join(__dirname, 'views'));

app.use(cors({
  origin: process.env.CORS_ORIGIN || 'https://ultimotradingltd.co.ke',
  credentials: true
}));

app.use((req, res, next) => {
  res.locals.basePath = BASE_PATH;
  next();
